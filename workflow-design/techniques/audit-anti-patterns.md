## Capability

Scan a workflow's content against the anti-pattern catalog, recording file, content, and recommended fix for each finding, and applying the per-anti-pattern detection heuristics below.

## Protocol

### 1. Walk The Catalog

- Walk every anti-pattern in [anti-patterns](../resources/anti-patterns.md); for each finding, record file, content, and recommended fix
- Common-input inheritance (AP-52): across each workflow, look for the same input declared on many techniques (watch for id drift — `planning-folder` / `planning-folder-path` / `planning_folder_path` are the same concept). Flag it for hoisting to the workflow-root (or group) `TECHNIQUE.md` under one canonical id, with the per-technique declarations removed and references resolving `{id}` via inheritance. Also flag path-flavored ids (a `-path` suffix). Do not flag a niche input shared by only two or three techniques
- Reference resolvability (AP-51): read every step of every `## Protocol` block and, for every referential phrase, confirm it resolves to a concrete target via the matching anchor form — input/output `{id}`, rule (dotted symbol), technique/op (`::` hyperlink), or resource (markdown hyperlink to a file in the workflow's `resources/`). Flag any bare prose reference that names a real target but is not anchored — most commonly a resource noun (e.g. "Apply design framework" where `design-framework.md` exists). Treat a reference DISGUISED as backtick code (`` `target_path` ``) or an angle placeholder (`<files>`) exactly as you would a bare one — if the token is a declared id, it must be braced; the wrapper is not a designator and does not resolve (AP-49 mode c). Do not be lulled by the markup looking intentional. Do NOT flag ordinary domain prose that names no formal artifact, anaphora already linked once in the step, or a reference whose target does not exist (that is a separate dangling-reference defect). This is the umbrella check; AP-42/46/48/49/50/58 are its per-kind specializations
- Protocol-variable binding hygiene (AP-58): for each technique, collect its protocol variables — the `{$name}` bindings and the bare `{name}` reads. Flag an UNBOUND LOCAL — a `{name}` read that matches no declared input/output (nor an ambient activity input such as `{target_path}` / `{branch_name}`) and has no `{$name}` binding in the protocol (the produce is missing — often a step computes the value in prose but never names it). Flag a DEAD BINDING — a `{$name}` never read as `{name}`. Confirm each binding textually precedes its reads; a value produced in mutually-exclusive branches legitimately carries `{$name}` in each producing branch, which is correct, not a double-bind
- Code-token backticking (AP-59): every literal code-like token in rendered prose — designators (`{id}` / `{id}.field`, `{$name}` / `{name}`), dotted rule addresses `technique.rule-name`, CLI/shell commands (including 'single-quoted' ones), MCP tool calls, resource URIs (`scheme://…`), and literal paths/filenames — is wrapped in backticks. Flag any bare-in-prose code token NOT already inside a backtick code span and NOT a markdown hyperlink target. Flag a 'single-quoted' command (convert the quotes to backticks). Flag a designator-containing token fragmented into adjacent spans (`` `git -C` `{x}` `…` ``) — it must be ONE span with the braces inside. Flag any surviving `{\$name}` escape — obsolete now that the backticked sigil is math-exempt; de-escape to `` `{$name}` ``. Do NOT flag descriptive prose nouns ("the planning folder"), only literal verbatim tokens
- Naming structure (AP-60): an identifier's grammatical shape must encode its kind. (1) BOOLEAN — flag a NON-AFFIRMATIVE boolean: a negated stem (`not_ready`, `no_merge`), a generic-noun id that buries the predicate (`…_flag`, `…_status`, `…_check`), or an ambiguous noun. Recommend the affirmative predicate (`squash_merge_supported`, `index_fresh`). The `is_`/`has_`/`can_`/`should_` prefix is VALUE-GATED — do NOT flag a conformant unprefixed affirmative boolean or a past-participle result flag (`worktree_created`, `review_passed`, `validation_passed`, `*_confirmed`); these already state the condition that holds and re-prefixing them is wrong. The test is AFFIRMATIVE, not PREFIXED. (2) COLLECTION/MAP — flag a representation-suffixed collection (`*_list`/`*_array`/`*_collection`/`*_set` → drop the suffix, pluralize the item noun) and a singular id whose value is an iterated collection (pluralize); a key-addressed map is singular and names the mapping (`domain_to_range`). (3) I/O id — flag a direction-encoded id (recommend head-noun-last qualified noun phrase, direction being structural), a representation-suffixed id (a `-path`/`-list` proxy, deferring to AP-42/52), synonym drift for one concept (hoist to one canonical id), and a BARE SINGLE-WORD generic-noun id (`summary`, `artifact`, `coverage`, `state`, `result`, `prompt`) that names a category not a concept (recommend qualifying with the parent/concept — `completion_summary`, `composed_prompt`). Do NOT flag a legitimate kind suffix `_mode`/`_type` — that IS the head noun. Three exemptions to the bare-single-word check: a bare PLURAL collection item-noun (`tasks`, `failures`) is correct per (2); an id mirroring an external tool/MCP parameter or a cross-boundary JSON-schema field keeps the external spelling even when bare (`body`, `query`, `findings`); a kind/enum discriminator (`_type`/`_mode`/`kind`) is already qualified-by-concept. Also flag a NAME COLLISION within one technique — any two Inputs/Output identifier headings (top-level `###` ids or `####` sub-fields) sharing a name, whether a parent and its own child or two siblings (recommend qualifying the colliding heading by its role — `#### matrix_rows`, `#### start_here_filename`); the sole exemption is a value carried unchanged across the input/output boundary (a mutated state variable read then written, or a pass-through echo), which is one concept (input∩output) and keeps its one name in both sections. (4) RULE SLUG — flag a non-assertive rule slug: a bare negation (`do-not-review-unresolved`), process-narration, or one that names a prohibited state; recommend the positive invariant (`assumptions-resolved-before-review`) ONLY where it reads at least as clearly — leave a negation that carries irreplaceable clarity (`no-cargo-here`, `do-not-mask-flaky`, `never-resume`). CAVEAT (shape ≠ meaning): a shape-conformant boolean whose MEANING is inverted still passes this shape check — meaning is the author's responsibility, not the heuristic's
- Invocation argument syntax (AP-53): scan every `## Protocol` step for a technique/operation invocation (`::[op](path)` or a hyperlinked op reference) whose argument list is wrapped in curly braces (`::op {arg: value}`). Flag it for conversion to parentheses attached to the op reference (`::op(arg: value)`), with variable/input argument values kept in designator braces (`(name: {$symbol})`) and literals left bare. Do NOT flag brace-with-colon forms that are not invocation arguments: Cypher node-property syntax inside a query string, JSON / output-template slots, and raw MCP-tool call documentation using object-arg syntax
- Unescaped `$` in prose (AP-54): scan for any `$` in rendered prose — outside fenced code blocks and inline code spans — and flag it for backslash-escaping (`\$`). Most commonly a bare-prose protocol-local designator (`{$name}`) or a literal `$` (prices, `$schema`, a `${VAR}` shown in running text). Do NOT flag `$` inside a code span or fenced block (CI/CD `${{ }}` content, backticked shell commands) — escaping there would corrupt the code
- Identifier case convention (AP-55): two namespaces, two cases. SYMBOLS (inputs, outputs, sub-fields, `{$locals}`) become runtime variables resolved by exact string match, so they MUST be `snake_case` to bind to the snake activity/condition/session-state namespace — flag any kebab/camel symbol id for conversion to snake (`target-path` → `target_path`), EXCEPT tool-param mirrors which keep the external tool's exact spelling (`session_index`, `cloudId`). NAMES (technique/operation/resource identities and their file / hyperlink / `::` targets, AND rule names — cited by dotted address, never evaluated) stay `kebab-case` — do NOT snake those
- Local-constraint note style (AP-56): scan every `## Protocol` step for an indented sub-bullet (`  - …`) that is a step-scoped CONSTRAINT — a conditional caveat, fallback, error-path, or prohibition qualifying its parent instruction (commonly "If …", "only when …", "do NOT … here"). Flag it for conversion to a `> ` blockquote note beneath the primary bullet (the parser otherwise flattens the sub-bullet into a disconnected peer step). Do NOT flag a genuine enumeration / sequential sub-step (e.g. per-harness branch tables) — those stay sub-bullets. Separately, flag any `## Rules` entry that in fact scopes to a single protocol block as a candidate to migrate DOWN to that block as a `> ` note (a global rule that is really local)
- Repeated/shared literal path (AP-57): scan for a filesystem path literal that appears more than once in a technique, or is shared across techniques, or that a declared variable already names (e.g. a step hard-coding `.engineering/artifacts/planning/…` when `{planning_folder_path}` exists). Flag for factoring into a defaulted input (fixed location) or `{$local}` (derived), referenced by designator. Do NOT flag a single-use literal, nor the one canonical statement (an input default, the producer step, or the artifact-location rule). If the literal puts an artifact OUTSIDE `{planning_folder_path}` against the artifact-location rule, flag it as a location defect (route to the variable), not merely a factoring one. Confirm case never encodes input-vs-output direction (that is structural — the `## Inputs`/`## Output` section); a value that is one technique's output and another's input is one snake symbol declared in both sections
- Rule symbol reference (AP-50): for every technique, scan `## Protocol` for rule citations written as prose ("per the X rule", "following the X rule") or with the `::` executable form, instead of the dotted symbol address `[workflow.]technique.rule` (shortened in-ancestry to the bare rule name). Also flag citations of a rule that is not declared anywhere (dangling). Recommend the dotted symbol address, or — for a dangling citation — a reference to the real inline content
- Designator bracing (AP-49): for every technique, scan `## Protocol` for references to a declared input/output (or a value of one) written without the brace designator — a bare identifier (`target-path`, `problem_statement`) or an orphan value (a bare index/enum value like "index 23"). Flag each; recommend bracing as `{declared_id}` (normalized to the declared spelling) or, for orphan values, naming the input the value belongs to ("when {declared_id} is 23"). Do NOT flag ordinary English words that merely coincide with an id
- Protocol self-description (AP-47): for every technique except the workflow-engine group, scan `## Protocol` for sentences describing the delivery mechanism rather than an action — "Resources are attached to technique responses", "available in the `_resources` field", "loaded via `get_technique`", "in the technique response", "the worker self-bootstraps". Flag each; recommend deleting the mechanism narration and keeping only the imperative action plus the canonical resource/technique reference
- Raw tool reference (AP-48): for every technique except the operation wrappers (`gitnexus-operations`, `harness-compat`), scan `## Protocol` for raw tool names that invoke a capability another technique wraps — e.g. `gitnexus_context`, `gitnexus_impact`, `gitnexus_cypher`, `gitnexus_query`, `gitnexus_list_repos`. Flag each; recommend the canonical hyperlinked reference to the wrapping op (e.g. `[gitnexus-operations](path)::[context](path)`), preserving arguments
- Resource→caller coupling (AP-44): for every resource (under `resources/`), check it does NOT name or link a technique that produces/consumes/calls it — backlink prose ("produced by", "composed by", "used by", "consumed by" + a technique name), markdown links to `../techniques/...`, or "Used By" / "Referenced By" catalogue columns mapping the resource to its consumers. Forward references (a prompt resource directing its reader to execute a technique, "use technique X" cross-refs, ontology/model descriptions) are NOT violations. Flag reverse caller-references; recommend describing the resource by what it is
- Artifact-name contract (AP-43): check that every artifact a technique creates or consumes carries its concrete name in the I/O declaration — a literal, a `{token}`-template, or a discriminator-keyed note — and NOT hardcoded in a Protocol step; and that a technique consuming multiple artifacts declares them as individually-named Inputs, not one opaque `*-paths` array. Flag Protocol steps that embed a filename (literal or `{templated}`) the I/O declaration should carry, and opaque multi-artifact path-array inputs
- Canonical artifact reference (AP-42): for every technique, check that `## Protocol` references artifacts by their canonical Input/Output identifier, not a literal filename or path (`assumptions-log`, not `assumptions-log.md`); that Input/Output ids are canonical names rather than path-flavored proxies (a `-path` suffix is the signal); and that an artifact built from a template hyperlinks its noun to the template's resource section (`[noun](../resources/<id>.md#section)`). Flag literal-filename references in Protocol, `-path` input/output ids, and template-backed artifacts whose noun is not linked to its resource section
- Unresolved output reference (AP-46): for every technique, check that every Protocol step referencing an output names the specific output designator (`{output_id}` or `{output_id}.field`) — never "the output", "the result", "the artifact", or any other vague pronoun. Each step must be anchored to a declared `## Output` entry. Flag any step where the reader must infer which output is meant
- Redundant label + link (AP-45): scan every technique for occurrences of `word ([same-word](url))` where the plain-text word and the link display text are the same name (modulo case/hyphen). Flag each occurrence and recommend collapsing to a single `[word](url)` hyperlink. Exclude cases where the preceding label is semantically distinct from the link text (e.g. a role name preceding a resource id)
- I/O coupling (AP-41): for every technique, inspect its `## Inputs` and `## Output`/`## Outputs` sections — flag any entry that names or links a workflow-internal producer or consumer of the value (another technique, an activity, a step, a checkpoint, a loop, or a workflow/activity file) as its source or destination. Markdown links to another technique `.md` and prose forms ("from X", "produced by X", "consumed by X", "for X") are the signal. Exclude `## Protocol`/`## Capability` utilisation references (including an inline "apply technique X" error recovery), resource links, and descriptions of a value's intrinsic/external nature ("git diff output", "the user's request", "provided by the server"). For each finding, recommend rewriting the entry to describe the value generically and removing the source/destination naming
- Authored artifacts[] (AP-65): for every activity, flag the presence of ANY `artifacts[]` block. Activities do not declare artifacts; the contract is synthesized by the server from the `## Outputs` of the techniques the steps bind (the technique outputs own artifact identity, AP-43). Recommend deleting the `artifacts[]` block; if a produced file then no longer appears in the synthesized contract, the producing technique under-declares — add a `#### artifact` to that technique's output (never back onto the activity). A non-file side effect (commit, PR) is not an artifact and is simply not declared. (The deterministic guard is scripts/check-artifact-description.ts.)
- Outcome states value not mechanics (AP-66): for every activity, scan `outcome[]` entries for mechanical delivery statements — "`<file>` written/created", a re-listing of an artifact's contents (owned by the producing technique's `## Outputs`), or "`<var>` populated/set". Flag each; recommend rewriting as the VALUE delivered (the capability unlocked, decision enabled, or state a consumer can now rely on), applying the rename test: would the outcome still read true and useful if its file/variable were renamed? Delete pure-plumbing outcomes ("directory created"). Do NOT flag an outcome already framed as delivered value
- Externalized technique output (AP-67): for every activity step that BINDS a technique, flag a `set` action whose `target` is the bound technique's own product (a value the technique computes — an assessment boolean, a classification, a derived structure, or a captured artifact path). Recommend declaring `### <target>` in that technique's `## Outputs` (same id, so it lands under the same bag name) + folding the `set` description into the `## Protocol`, then deleting the `set`. Do NOT flag: a cross-iteration accumulator gather (appends per-iteration outputs across a `forEach` loop — scatter-gather), a caller-specific value derived from a generic tool-wrapper op (AP-41 coupling), or a `set` on a control step (no `technique`)
