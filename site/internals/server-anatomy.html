Server anatomy

The architecture models describe what the server guarantees; this page describes the code that delivers it — roughly 5,400 lines of TypeScript in src/, organized around two tool registrars, a family of loaders, and a session store. It is the map for reading the source.

This page and the rest of the internals section are derived from the server source itself; each section links to the files it describes. The contributor's build-and-test guide is docs/development.md.

The shape of the codebase

Startup is a short, linear path: the entry point parses configuration, a factory builds the MCP server, and two registrar functions attach all sixteen tools as closures. There is no routing layer and no dependency-injection framework — the registrars close over the ServerConfig and call shared machinery directly.

Module map

Each module owns one concern. The table lists them in the order a request touches them:

ModuleResponsibility
index.ts, config.ts, server.tsEntry point, ServerConfig (workflow directory, schemas directory, workspace root from --workspace or WORKFLOW_WORKSPACE), and the server factory.
tools/The sixteen tool handlers. Navigation, checkpoint, and trace tools live in workflow-tools.ts; session creation and content fetching live in resource-tools.ts.
loaders/Reading definitions: workflow and activity YAML (workflow-loader.ts), technique resolution and composition (technique-loader.ts), markdown technique parsing (markdown-technique-loader.ts), resources (resource-loader.ts), and the generated JSON schemas (schema-loader.ts).
schema/Zod definitions for every file shape the server reads or writes — workflow, activity, condition, technique, resource, session — plus identifier rules. The JSON schemas in schemas/ are generated from these sources.
utils/session/The session store: planning-folder resolution, atomic session.json writes, HMAC sealing (crypto.ts), deterministic session-index derivation (derivation.ts), legacy-layout migration (migration.ts), and the load/save resolver every tool uses.
utils/validation.tsThe runtime fidelity validators: activity transitions, workflow version drift, step and activity manifests, transition conditions, and technique-fetch cross-checks.
utils/delivery.ts, utils/binding-provenance.tsReference-not-repeat delivery (content hashing, the per-agent delivered ledger, unchanged markers) and the static provenance annotations on technique inputs and outputs.
trace.ts, logging.tsThe in-process TraceStore (per-session event buffers, evicting the oldest beyond 1,000 sessions), HMAC-signed trace tokens, and the withAuditLog wrapper around every handler.
result.ts, errors.tsThe Result<T, E> type loaders return, and the typed domain errors (below).

The layer every call passes through

Three functions form a choke point that every authenticated tool call traverses, which is what makes the server's observability and integrity guarantees uniform rather than per-tool:

  1. withAuditLog (logging.ts) wraps each handler — it times the call, emits a structured audit event to stderr, and appends a trace event to the session's TraceStore buffer.
  2. loadSessionForTool (session/resolver.ts) resolves the session_index to a planning folder, verifies the HMAC seal, and validates session.json against the session schema before the handler sees any state.
  3. saveSessionForTool persists the updated state atomically and recomputes the seal, so no code path can write state without resealing it.

Everything cryptographic — session seals, session indexes, trace tokens — funnels to a single 32-byte server key created lazily at ~/.workflow-server/secret (file mode 0600) by session/crypto.ts. One key, three uses, one place to reason about it.

Error discipline

Loaders return Result<T, E> values rather than throwing, and failures that must cross the tool boundary are typed so handlers can map them to user-facing responses without string matching:

ErrorRaised when
WorkflowNotFoundError, ActivityNotFoundError, TechniqueNotFoundError, ResourceNotFoundErrorA definition referenced by id does not exist in the workflow directory.
WorkflowValidationErrorA definition file parses but fails schema validation; the error carries the issue list.
SessionStoreErrorSession state cannot be loaded or written. The code field distinguishes NOT_FOUND, COLLISION, SEAL_MISMATCH, INVALID_INDEX, and WORKSPACE_INVALID.
MarkdownTechniqueParseError, MigrationError, WorkspaceConfigErrorA technique file violates the technique protocol, a legacy planning folder cannot be upgraded, or the workspace path is unusable at startup.

How these surface to a calling agent — and how they differ from non-blocking validation warnings — is covered in the protocol in practice.

Where things live on disk

  • Workflow definitions are read from the configured workflow directory — in this repository, the workflows/ worktree of the orphan workflows branch (why an orphan branch).
  • Session state is written under <workspace>/.engineering/artifacts/planning/<slug>/ — the session store page covers the folder's contents.
  • The server key lives at ~/.workflow-server/secret, per user, outside any repository.
