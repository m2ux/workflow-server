### Usage Patterns

**How the seam is used today:** `next_activity` is the orchestrator's transition tool, called once per activity transition. Its mutator (`workflow-tools.ts:434-456`) is the single place where `activity_exited` / `activity_entered` / `workflow_completed` events are emitted. Every state mutation runs through `advanceSession(state, draft => …)` → `saveSessionForTool`, which bumps `seq`/`ts`, deep-clones a draft, runs the mutator, re-seals, and atomic-writes.

**Call frequency:** once per activity boundary; workflow-completion path fires once at the terminal activity (`activity_id === 'complete'` or `TERMINAL_SENTINEL`, `:452`); parent-notification fires once per child on child terminal (`:463`).

### Dependencies

**Depends On (inbound to the feature):**
- The calling **orchestrator/harness** to relay native usage figures — the server cannot self-measure (RS-3, confirmed: the agent cannot read its own mid-turn usage; the Messages API `usage` is post-hoc). This is a hard external dependency the workflow/technique instruction must drive.
- A config **price table** for cost (new; mirrors `config.ts:49-50`).

**Depended On By (what the feature touches):**
- `session.json` shape (`SessionFile`) → consumed by `inspect_session`, `get_workflow_status`, `resolver.sessionView`, and the migration path. An **optional-with-absent** field keeps all existing readers and all existing `session.json` files parsing unchanged.
- Generated JSON-Schema mirrors `schemas/session-file.schema.json`, `schemas/state.schema.json` → regenerated by `build:schemas`; asserted in sync by `tests/generated-schemas.test.ts`.

### Architecture

**Existing patterns the feature reuses (no new machinery required):**
- **Per-event history + rolled-up field.** The codebase already separates the append-only `history[]` (per-event audit) from rolled-up state (`variables`, `completedActivities`). Usage maps the same way: a `usage_recorded` history event (audit trail) + an optional rolled-up `SessionFile.usage` field (fast read at completion) — mirroring how `technique_bundled` carries `{ techniqueId, stepId, agentId }` in open `data`.
- **Caller-declared token quantity as a Zod param.** `get_activity` already accepts `context_tokens` (`params.ts:25-28`, consumed `workflow-tools.ts:638`). The `usage` param is the same shape, one level richer (an object of counts rather than a scalar).
- **Config constant + env override + in-code fallback.** `bundleHeadroomFraction`/`bundleCharsPerToken` (`config.ts:49-50`, read via `envNumberOrDefault`, `:62-67`) is the house style for a tunable numeric-policy knob; the price table + `priceTableVersion` follow it.
- **Recursive single-file tree.** The whole work-package tree lives in one top-level `session.json` via `triggeredWorkflows[i].state` (a full embedded `SessionFile`, `session.schema.ts:35`, `:120`, `:213`). Per-workflow roll-up that includes children is a pure traversal of this in-memory tree — no cross-file reads.

**Known technical debt bearing on the feature:**
- Two parallel state models coexist: live `SessionFile` (`session.schema.ts`) and legacy `WorkflowState` (`state.schema.ts:114-147`, migration-only). The feature targets `SessionFile`; both import `HistoryEntrySchema`, so the new event type is shared automatically, but the legacy `WorkflowState` needs no `usage` field (it only surfaces via `migration.ts`).

### The server writes no `.md` artifacts (decisive for SC-1)

The server's **only** file-write surface is the sealed session store: `writeSessionFile` (`store.ts`) writes `session.json` + `.session-token` via `writeAtomic` (tmp + fsync + rename, `store.ts:182-245`). A repository-wide search for `.md`/`writeFile` in `src/tools/` and `src/loaders/` finds **no** artifact-writing code — the server never produces planning documents. Every planning `.md` (including the folder `README.md` progress table and the close-out `COMPLETE.md`) is written by the **worker/orchestrator agent** executing a workflow technique (e.g. `finalize-documentation::create-complete-doc`, the `finalize-activity` README-update step).

**Implication for the `NN-token-usage.md` artifact (SC-1):** the server produces and persists the *structured* usage/cost record inside `session.json` (history events + rolled-up field); the **agent renders it to `NN-token-usage.md` + the README summary/link** as a workflow-technique step, reading the figures out of session state. This split matches the existing division of labour (server owns durable structured state; agent owns human-facing artifacts) and avoids giving the server a filesystem-writing role it does not have today. Rendering can read the roll-up either via a new field surfaced through `inspect_session` / `get_workflow_status`, or by scanning `usage_recorded` history events — the plan should pick one read path.

## Concrete Extension-Point Specification

The seven analysis targets, each grounded in code. This is the buildable core the plan expands into tasks.

### 1. Channel binding — the `usage` param on `next_activity`

Add one optional Zod param to the `next_activity` input object (`workflow-tools.ts:374-380`), alongside `step_manifest` / `activity_manifest`:

```ts
// hoisted like stepManifestSchema (workflow-tools.ts:34-43)
const usageSchema = z.object({
  input_tokens: z.number().int().nonnegative(),
  output_tokens: z.number().int().nonnegative(),
  total_tokens: z.number().int().nonnegative().optional(),   // derivable = input+output; accept if relayed
  cache_read_tokens: z.number().int().nonnegative().optional(),
  cache_write_5m_tokens: z.number().int().nonnegative().optional(),
  cache_write_1h_tokens: z.number().int().nonnegative().optional(),
  model: z.string().min(1).optional(),                       // model id for pricing/stamp
}).optional().describe('Native model usage for the activity being EXITED, relayed by the orchestrator/harness (the worker cannot self-measure). Omit entirely when no usage is available — the run still completes. Absent sub-figures are omitted, not fabricated.');
```

- **Declaration + validation site:** the input-schema object at `workflow-tools.ts:374-380`; Zod validates it at the MCP boundary exactly as `context_tokens` is validated in `get_activity`. Malformed → fails loudly at the boundary; absent → `undefined`, degrades gracefully (SC-6). Destructure it in the handler signature at `:381`.
- **Optionality + graceful degradation:** `.optional()` means existing callers (and any harness that relays nothing) are unaffected — SC-6, backward compatibility. Sub-figures are individually optional so a harness reporting no cache figures omits them (SC-2). `total_tokens` is accepted-if-present, else derived `input + output`.
- **Attribution semantics:** the figure describes the activity being **exited** (`draft.currentActivity` at `:438`), consistent with per-activity granularity at the transition seam (RE-3). Finer attribution (checkpoint/dispatch windows) is deferred (DI-2). This is the known "attribution-to-exited-activity" risk (see Risks) — accepted for v1.
- **Workflow/technique instruction obligation:** because usage is **orchestrator/harness-relayed, never worker-self-measured** (RS-3), a workflow/technique instruction must direct the orchestrator to read its harness usage figure and pass it as `usage` on each `next_activity` call. This is a **workflow-corpus change** (a technique/rule addition), separate from the server change — the plan must scope it explicitly. The server change alone is inert without the corpus instruction that populates the param.

### 2. Session schema — the `usage` record + `usage_recorded` event

**(a) New history event type.** Add one member to `HistoryEventTypeSchema` (`state.schema.ts:6-30`):

```ts
'usage_recorded',   // data: { activityId, input_tokens, output_tokens, total_tokens, cache_*, model, cost_usd|null, priceTableVersion }
```

`HistoryEntry.data` is already open (`z.record(z.unknown())`, `:41`), so the payload needs **no** further schema work — the exact pattern `technique_bundled` uses. This gives the per-event audit trail.

**(b) Rolled-up field on `SessionFile`.** Add an optional field to `SessionFileBaseSchema` (`session.schema.ts:57-157`) and the `SessionFile` interface (`:163-187`):

```ts
usage: z.object({
  perActivity: z.record(z.object({
    input_tokens: z.number().int().nonnegative(),
    output_tokens: z.number().int().nonnegative(),
    total_tokens: z.number().int().nonnegative(),
    cache_read_tokens: z.number().int().nonnegative().optional(),
    cache_write_5m_tokens: z.number().int().nonnegative().optional(),
    cache_write_1h_tokens: z.number().int().nonnegative().optional(),
    model: z.string().optional(),
    cost_usd: z.number().nonnegative().nullable(),          // null = unknown/unpriced model (SC-4)
    priceTableVersion: z.string().optional(),
  })).default({}),
  workflowTotal: z.object({
    input_tokens: z.number().int().nonnegative(),
    output_tokens: z.number().int().nonnegative(),
    total_tokens: z.number().int().nonnegative(),
    cost_usd: z.number().nonnegative().nullable(),          // null if any activity is unpriced
  }),
}).optional(),   // optional-with-absent → existing session.json parse unchanged
```

Add the matching field to the `SessionFile` **interface** (`:163-187`) — the interface is hand-maintained alongside the Zod schema (the recursive-schema pattern requires it).

**(c) `TOP_LEVEL_KEY_PRIORITY` placement.** Add `'usage'` to the array (`store.ts:85-107`) — a natural slot is after `variables`/`checkpointResponses` and before `history` (rolled-up state near the other rolled-up state, before the event log). Omitting it is not a correctness bug (unlisted keys sort lexicographically after the priority block) but the convention lists every top-level field.

**(d) Optional seeding.** If the roll-up should exist empty from creation, initialize it in `createInitialSessionFile` (`session.schema.ts:258-302`) alongside the `history` seeding. Leaving it absent-until-first-usage is equally valid and slightly leaner (matches how `deliveredContent` stays absent until first use).

**Invariants preserved:** all writes still flow through `advanceSession` → `saveSessionForTool` → `writeSessionFile`, which re-canonicalises, re-HMAC-seals, and atomic-writes. Because agents cannot write `session.json` directly (the `.session-token` seal, `store.ts` `verifySeal`, rejects out-of-band edits), only the server writes usage — via the standard mutate path. Zod validation and HMAC-seal invariants are untouched.

### 3. Aggregation — capture + roll-up at the `next_activity` seam

All aggregation lives inside the existing `advanceSession` mutator (`workflow-tools.ts:434-456`), extended:

- **Capture (per-activity):** in the exit-prior block (`:438-443`), when `draft.currentActivity` is non-empty **and** the `usage` param was supplied, compute `cost_usd` (§4), push a `usage_recorded` history event keyed to `draft.currentActivity`, and write/merge `draft.usage.perActivity[draft.currentActivity]`. Merge (not overwrite) so a re-entered activity accumulates.
- **Roll-up (per-workflow):** maintain `draft.usage.workflowTotal` incrementally as each per-activity figure lands (cheap, no history re-scan). `cost_usd` on the total is `null` if any contributing activity is unpriced (SC-4 semantics propagate to the total).
- **Child (`triggeredWorkflows`) roll-up (SC-5):** the whole tree is in one `session.json` via `triggeredWorkflows[i].state` (recursive `SessionFile`). Two mechanisms combine:
  1. **In-tree traversal at completion:** at the workflow-completion block (`:452-455`), when computing the final `workflowTotal`, recurse `draft.triggeredWorkflows[]` into each `.state`, summing each child's `state.usage.workflowTotal` into the parent total. A small pure helper `sumUsageTree(state): WorkflowTotal` that walks `state.usage.workflowTotal + Σ sumUsageTree(child.state)` gives the child-inclusive figure. This is pure computation over the in-memory tree — no extra I/O.
  2. **Parent-notification path (`:463-485`):** when a child reaches terminal, the parent's `triggeredWorkflows[i].status` already flips to `completed` in a second `advanceSession` cycle against the parent. This is the natural place to also fold the child's `usage.workflowTotal` into the parent's aggregate if the parent should reflect child spend *before its own completion*. For v1, the completion-time in-tree traversal (mechanism 1) is sufficient and simpler — the parent's rendered artifact reads the child-inclusive total at completion. The plan should pick one (recommend mechanism 1 for v1; note mechanism 2 as the incremental-visibility alternative).

### 4. Cost model — config price table, frozen at activity time

Mirror the bundling-knob idiom in `config.ts`:

- **Location/shape:** add to `ServerConfig` (`config.ts:4-34`) a `priceTable` (per-model `{ input, output }` USD-per-MTok) and a `priceTableVersion: string`. Provide `DEFAULT_PRICE_TABLE` + `DEFAULT_PRICE_TABLE_VERSION` constants next to `DEFAULT_BUNDLE_*` (`config.ts:49-50`), seeded from the pricing captured in [kb-research](04-kb-research.md#current-claude-pricing-captured-2026-07-15--seeds-the-config-price-table-re-confirm-at-build-time). Wire into `loadConfig` (`:125-135`) with an env override (e.g. `PRICE_TABLE_VERSION`) via `envOrDefault`; the table itself is a build-time constant (a full-table env override is over-engineering for v1 — a micro-decision below).
- **Derived cache rates (RS-2):** store only base `{ input, output }` per model; derive 5m-write ×1.25, 1h-write ×2, cache-read ×0.1 from the universal multipliers. Keeps the table small and cache pricing correct by construction.
- **Freeze-at-activity-time (SC-3, RE-2):** cost is computed in the `next_activity` mutator at the moment the activity exits, from the table then in effect, and written into the `usage_recorded` event + `perActivity` entry stamped with `model` + `priceTableVersion`. It is never recomputed at render time (out-of-scope #3) — the recorded number is the record.
- **Unknown-model degradation (SC-4):** if `usage.model` is absent from the table, record all token figures and set `cost_usd: null` (rendered `unknown`); the run never blocks. A missing `model` on the param behaves identically (tokens recorded, cost `null`).
- **Cost basis:** compute per figure: `cost = input×rate.input + output×rate.output + cache_read×(rate.input×0.1) + cache_write_5m×(rate.input×1.25) + cache_write_1h×(rate.input×2)`, all divided by 1e6 (per-MTok). A pure `estimateCost(usage, priceTable, version)` helper (unit-testable in isolation) is the clean home.

### 5. Artifact writing on completion — server persists structured, agent renders

Per the Implementation Review finding, **the server writes no `.md`**. The division:

- **Server (this feature's server change):** persists the structured usage/cost record in `session.json` — `usage_recorded` history events + the rolled-up `usage` field, child-inclusive at completion. Surface it for reading through the existing read tools (`inspect_session` / `get_workflow_status` already project `SessionFile`) so the renderer can fetch it.
- **Agent (a workflow-corpus change):** renders `NN-token-usage.md` (per-activity token table + per-workflow totals + cost estimate) and the one-line README summary/link, as a step in the completion path — the same place `finalize-documentation::create-complete-doc` and the `finalize-activity` README-update step already write human-facing artifacts. This is a **second corpus change** the plan must scope (alongside the §1 orchestrator-populates-`usage` instruction). Label the recorded cost an **estimate** and carry the API-key-vs-subscription caveat into the artifact/README (kb-research risk row).

## Risks and Open Micro-Decisions (for plan-prepare)

**Risks:**
- **Relayed-figure authority (convergent lens finding):** a wrong `usage` figure silently corrupts a durable cost record (unlike a wrong `context_tokens`, which only mis-sizes a bundle and self-corrects). Mitigation already in the design: the declared, Zod-validated param keeps a *missing* figure visibly missing (SC-6), and every figure is stamped with provenance (`model`, `priceTableVersion`). No server-side fabrication.
- **Attribution-to-exited-activity (pedagogy lens):** tokens spent during the checkpoint-yield / dispatch window between transitions are mis-filed onto the exited activity; the workflow total still sums correctly, so the error is arithmetically invisible. Accepted for v1 (RE-3); finer attribution deferred (DI-2).
- **Cost is an estimate, not billing:** client-side estimate; meaningless on Pro/Max subscriptions. Mitigation: label as estimate + carry the caveat into the artifact/README (kb-research).
- **Two corpus changes are load-bearing:** the server param is inert without (a) the orchestrator-populates-`usage` instruction (§1/G6) and (b) the completion renderer (§5/G5). The plan must scope both as first-class tasks, not server-only.

**Open micro-decisions for plan-prepare:**
1. **Roll-up read path for the renderer:** re-scan `usage_recorded` history events, or read the rolled-up `SessionFile.usage` field via `inspect_session`/`get_workflow_status`? (Recommend the rolled-up field — one read, already child-inclusive.)
2. **Child roll-up timing:** completion-time in-tree traversal (`sumUsageTree`, recommended for v1) vs. incremental fold at the parent-notification path (`:463-485`). Pick one.
3. **Seed empty `usage` at creation** (`createInitialSessionFile`) vs. absent-until-first-use (recommend absent-until-first-use, matching `deliveredContent`).
4. **`total_tokens`:** accept-if-relayed vs. always-derive (`input+output`). (Recommend derive; accept relayed as a cross-check.)
5. **Price-table override surface:** version-string env override only (recommended) vs. a full-table env/file override (over-engineering for v1).
6. **Where the cost/roll-up helpers live:** new `src/utils/usage.ts` vs. co-located in the schema/handler. (Recommend a dedicated `src/utils/usage.ts` for isolated unit tests.)
