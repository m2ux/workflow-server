# Session inspection tool — remove ad-hoc `session.json` introspection

## Status

Proposal. Captures a friction source observed during a live work-package (review-mode)
run and proposes a first-class remedy in the workflow-server.

## Problem

Close-out activities need to read a session's own state back out of `session.json` —
the variable bag, the checkpoint responses, the completed/skipped activity lists, the
history event stream, and (for the meta session) the embedded child session under
`triggeredWorkflows[].state`. The workflow-server has no first-class way for a worker
to obtain that state, so workers fall back to inline shell:

```
python3 -c "import json; d=json.load(open('.../session.json')); ..."
```

In one meta + client run this happened eight times, split across two activities:

| Activity (technique) | What the ad-hoc call read |
|---|---|
| `complete` (`conduct-retrospective`) | variable bag keys; history event-type counts; checkpoint/activity events |
| `end-workflow` (`verify-outcomes`, `generate-summary`) | top-level structure; identity + activity lists; `triggeredWorkflows[0].state` child trace; meta variable bag |

Four of the eight were pure structure-probing — the worker did not know where a field
lived (`variables` at top level vs `state.variables` in a child, `type` vs `event` on
history entries) and iterated to find it.

## Why this is friction

- Every inline `python3 -c "..."` is a distinct command string, so it triggers a
  permission prompt, and "don't ask again" never helps because the next call's body
  differs.
- The local `allow-project-scripts` PreToolUse hook deliberately auto-allows only a
  path-invoked project *file* (`python3 file.py`); it refuses `python3 -c` by design.
  So the inline form can never be allow-listed.
- The logic is regenerated by the model each time and has to re-discover the schema,
  which is both wasteful and a correctness risk (a probe that guesses the wrong key
  silently reports nothing).

## Decision

Add a read-only introspection **tool** to the workflow-server and have the close-out
**techniques advise the worker to call it**. A tool call is preferred over shipping
static scripts because:

- It is not Bash — no permission prompt, on any harness, regardless of worker cwd or
  worktree location.
- It is server-owned and versioned with the session schema, so the projection logic
  tracks the schema and never guesses.
- The server already resolves `session_index` → the on-disk `session.json`, so the
  worker supplies only the index and a view name — no path handling, no file reads.

### Proposed tool

`inspect_session` — returns a compact, structured projection of a session's state.

| Param | Type | Notes |
|---|---|---|
| `session_index` | string (required) | The 6-char index; server resolves it to `session.json`. |
| `view` | enum (optional, default `summary`) | `summary` \| `identity` \| `variables` \| `checkpoints` \| `activities` \| `history` \| `children`. |
| `child_index` | integer (optional) | Target an embedded `triggeredWorkflows[n]` child instead of the root session. |
| `variable` | string (optional) | With `view=variables`, return just this one key. |

Returns structured data (not the raw file): for `summary`, the identity block
(`workflowId`, `sessionIndex`, `status`, `currentActivity`, `startedAt`), the
completed/skipped activity lists, the variable bag, the checkpoint responses
(`id → optionId @ respondedAt` plus any `variablesSet`), a history event-type tally,
and a one-line-per-child digest of `triggeredWorkflows`. Narrower views return just
their slice.

### Techniques that should advise it

The activities where the ad-hoc calls occurred:

- `workflow-engine::verify-outcomes` (meta `end-workflow`) — read the client session's
  status, completed activities, and variable bag to reconcile declared outcomes.
- `workflow-engine::generate-summary` (meta `end-workflow`) — read activity list,
  checkpoint responses, and the child trace to compose the summary.
- `conduct-retrospective::retrospective` / `::select-next` (client `complete`) — read
  the variable bag and checkpoint/activity history for the retrospective.

Each should add a step note: obtain session state via `inspect_session`, not by reading
`session.json` directly.

## Alternatives considered

- **Static script + allow-hook** (`scripts/inspect_session.py`, invoked by path so the
  `allow-project-scripts` hook approves it). Works, but is machine-local (depends on a
  user's hook), is still a Bash call, and is fragile to the worker's cwd/worktree. Kept
  as the reference implementation of the projection logic (see `scripts/`) and as a
  fallback for environments without the tool, but not the recommendation.
- **Return the whole `session.json`** from an existing tool. Rejected: the file is large
  (history + `deliveredContent`) and pushes the projection work back onto the worker,
  which is what caused the schema-probing in the first place.

## Reference implementation

`scripts/inspect_session.py` implements the exact projections above against the current
schema (`schemaVersion: 1`). It doubles as the behavioral spec for the tool's output
contract and as the fallback described under Alternatives.
