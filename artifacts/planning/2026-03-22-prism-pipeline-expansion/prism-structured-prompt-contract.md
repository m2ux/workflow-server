# Design: Structured Prompt Contract for Prism

> Callers pass intent. Prism decides everything.

---

## 1. The Contract

### What callers provide

A single prose variable — `analysis_prompt` — that describes what to understand, why, and any constraints. No prism configuration values (no `pipeline_mode`, no `selected_lenses`, no resource indices).

```
analysis_prompt: |
  Evaluate this proposal for internal consistency (do the parts
  agree with each other?), veracity of claims (are the cost
  projections in Section 4.2 supported by evidence?), and
  feasibility (do the resource constraints in Section 2.1
  actually support the delivery timeline in Section 5?).
  Prioritise accuracy over speed.

target: /path/to/proposal.md
output_path: /path/to/output/
```

The `analysis_prompt` can be:
- Provided directly by the caller or user
- Loaded from a previously-saved prompt artifact (re-use scenario)
- Generated by the elicitation step when absent (interactive scenario)

### What callers must NOT provide

| Variable | Status | Reason |
|---|---|---|
| `pipeline_mode` | Removed from caller contract | prism decides per-dimension |
| `selected_lenses` | Removed from caller contract | prism maps goals to lenses |
| `analysis_focus` | Removed from caller contract | prism derives from target content |
| `budget` | **Kept** — this is a constraint, not a decision | "prioritise accuracy" vs "minimise cost" |

### What prism decides

From the prompt + target content:
- How many analytical dimensions exist
- Which pipeline mode per dimension
- Which lenses per dimension
- Whether dimensions can be grouped
- Execution order (deepest-impact dimensions first)
- How to compose the report (single REPORT.md vs multi-dimensional EVALUATION-REPORT.md)

## 2. The `select-mode` Activity Flow

### Step sequence

```
select-mode:
  1. elicit-goals        ← ONLY when analysis_prompt is absent
  2. confirm-prompt      ← checkpoint: user reviews generated prompt
  3. parse-prompt        ← extract dimensions from the prompt
  4. detect-scope        ← classify the target
  5. map-dimensions      ← map dimensions to modes/lenses
  6. summarize           ← present the execution plan
  7. confirm-plan        ← checkpoint: user confirms or adjusts
  8. apply-plan          ← set analysis_units
```

### Elicitation step: `elicit-goals`

Runs ONLY when `analysis_prompt` is absent. Produces a structured prompt through 1-2 targeted questions, saves it as a reusable artifact, then continues execution in the same run.

```
elicit-goals:
  condition:
    type: simple
    variable: analysis_prompt
    operator: notExists

  steps:
    1. Ask: "What do you want to understand about this target?"
    2. Optionally ask: "What matters most — depth, breadth,
       accuracy, or cost?"
    3. Compose a structured analysis_prompt from the answers
    4. Write to {planning_folder}/analysis-prompt.md
    5. Set analysis_prompt variable from the composed text
```

**Gate:** Skipped when `analysis_prompt` is already set — whether provided by a caller, loaded from a saved prompt file, or typed inline by the user. This means:
- Callers passing a structured prompt → elicitation skipped
- Users loading a prior prompt file → elicitation skipped
- Vague requests with no prompt → elicitation runs, then execution continues

### Prompt artifact format

Saved to `{planning_folder}/analysis-prompt.md`. Plain markdown — readable, editable, versionable.

```markdown
# Analysis Prompt

## Target
/path/to/proposal.md

## Dimensions
- **Consistency**: Do the cost projections in Section 4.2 align
  with the resource constraints in Section 2.1?
- **Veracity**: Are the market size claims supported by cited
  evidence?
- **Feasibility**: Does the 6-month timeline account for the
  infrastructure dependencies listed in Section 3?

## Constraints
- Prioritise accuracy over speed
- Focus on Sections 2, 4, and 5
```

The user can edit this file and re-run prism with it later — adjusting dimensions, adding constraints, narrowing focus — without going through elicitation again. The prompt becomes a reusable, iteratable analysis specification.

### Prompt review checkpoint: `confirm-prompt`

After elicitation generates the prompt, the user reviews it before execution proceeds. This checkpoint only fires when elicitation ran (i.e., the prompt was generated, not provided).

```
confirm-prompt:
  condition:
    type: simple
    variable: prompt_was_elicited
    operator: "=="
    value: true

  message: "Here is the analysis prompt I've composed from your
    input. It has been saved to {planning_folder}/analysis-prompt.md.
    Review and confirm to proceed, or edit before continuing."
  options:
    - id: proceed
      label: "Proceed with this prompt"
    - id: edit
      label: "Let me refine the prompt"
      effect:
        transitionTo: select-mode  (re-enters elicitation)
```

When the user provides or loads a prompt directly, this checkpoint is skipped — no review is needed for a prompt the user already authored.

## 3. Changes to `plan-analysis`

### New protocol step: `parse-prompt`

Runs after elicitation (if needed) and after the prompt checkpoint. At this point `analysis_prompt` is guaranteed to exist.

```
parse-prompt[5]:
  - "Read analysis_prompt. If absent (backward-compatible
     fallback — no elicitation, no prompt), fall through to
     existing detect-scope logic."
  - "Extract analytical dimensions: look for evaluation axes,
     concerns, questions, or focus areas described in prose.
     Each distinct analytical question becomes a dimension.
     Single-concern prompts produce one dimension."
  - "For each dimension, extract: name (short label),
     description (what to examine), evidence-pointers
     (any specific target content referenced in the prompt),
     and budget signal (accuracy/speed/cost keywords)."
  - "Read the target (or a sample for large targets) to
     understand its structure, sections, claims, and
     architecture. Produce content-aware analysis_focus
     values per dimension that reference actual target
     content."
  - "Proceed to map-dimensions."
```

### New protocol step: `map-dimensions`

Replaces the current `query-recommendation` / `single-unit-recommendation` split for prompted analyses.

```
map-dimensions[4]:
  - "For each dimension, select a pipeline_mode using the
     mode-inference rules: consistency/coherence → full-prism
     or verified. truthfulness/claims → dispute or portfolio
     (claim + knowledge-audit). alternatives/viability →
     single (rejected-paths). resources/constraints → single
     (scarcity). accuracy-critical → verified. broad
     exploration → portfolio. self-correction needed → dispute.
     complex multi-class code → subsystem. unknown/delegated →
     smart."
  - "For each dimension, select lenses using the existing
     goal-mapping matrix. Compose analysis_focus from the
     dimension description + evidence-pointers + target
     content discovered during parse-prompt."
  - "Group dimensions that share a pipeline_mode AND target
     into execution groups where possible. Full-prism and
     verified dimensions cannot be grouped (they run the
     full pipeline). Portfolio dimensions with different
     lenses can be grouped into a single portfolio run."
  - "Build analysis_units: one unit per execution group.
     Each unit has: target, target_type, pipeline_mode,
     lenses, analysis_focus, dimension_names (which
     dimensions this unit covers), output_subdir."
```

### What happens to existing steps

| Existing step | Status |
|---|---|
| `detect-scope` | Kept — still needed for target classification |
| `query-recommendation` | Kept — fallback when `analysis_prompt` is absent |
| `single-unit-recommendation` | Kept — fallback when `analysis_prompt` is absent |
| `select-pipeline-mode` | Kept — fallback for keyword matching when no structured prompt |
| `survey-structure` | Kept — used for codebase scope |
| `classify-units` | Kept — used for codebase scope |
| `select-strategy-per-unit` | Modified — respects per-dimension modes from `map-dimensions` |
| `build-analysis-units` | Modified — accepts multi-mode units from `map-dimensions` |

The key property: **everything is backward-compatible.** If `analysis_prompt` is absent, the existing flow runs unchanged. The new steps only activate when a structured prompt is provided.

## 3. Multi-mode Execution Model

### The current problem

Today, `pipeline_mode` is a single global variable. `select-mode` transitions to ONE activity based on it. If analysis_units contains units with different modes, the transition can only go to one place.

### The solution: per-unit dispatch in an orchestration loop

Replace the current transition model:

```
select-mode → structural-pass → [adversarial → synthesis] → generate-report
```

With a dispatch loop:

```
select-mode → execute-plan (loop over units) → generate-report
                  ↓ per unit
            ┌─ structural-pass (single, full-prism, portfolio, behavioral)
            ├─ dispute-pass
            ├─ subsystem-pass
            ├─ verified-pass
            ├─ reflect-pass
            ├─ smart-pass
            └─ adaptive-pass
```

This requires a new activity: `execute-plan`.

### `execute-plan` activity

```
id: execute-plan
description: "Iterate over analysis_units and dispatch each unit
  to the appropriate pipeline activity based on its pipeline_mode.
  Each unit runs its full pipeline chain (e.g., structural →
  adversarial → synthesis for full-prism units). After all units
  complete, transition to generate-report."

loops:
  - id: unit-dispatch
    type: forEach
    variable: current_unit
    over: analysis_units
    steps:
      - id: set-unit-context
        description: "Set pipeline_mode, output_path, lenses
          from current_unit for the dispatched pipeline."
      - id: dispatch-pipeline
        description: "Dispatch the current unit to the activity
          matching its pipeline_mode. Each dispatch is a fresh
          worker (or chain of workers for multi-pass modes)."
      - id: collect-unit-artifacts
        description: "Collect artifact paths from the completed
          pipeline and append to all_artifact_paths."

transitions:
  - to: generate-report
    isDefault: true
```

The dispatch step routes to the right activity chain. For single-unit prompts (the common case), this loop executes once — identical performance to today.

### Backward compatibility

| Invocation | analysis_units | execute-plan behavior |
|---|---|---|
| Direct user, single file, no prompt | 1 unit (single L12) | Loop runs once, dispatches to structural-pass |
| Direct user, codebase scope | N units (per-module modes) | Loop runs N times, same as today's structural-pass loop |
| Structured prompt, 1 dimension | 1 unit | Loop runs once |
| Structured prompt, 4 dimensions | 4 units with different modes | Loop runs 4 times, each to different activity |
| prism-evaluate trigger (legacy) | 1 unit (mode pre-set) | Loop runs once — identical to today |

## 4. Report Generation

### Single-dimension analysis

When `analysis_units` has one entry, `generate-report` produces REPORT.md exactly as today.

### Multi-dimension analysis

When `analysis_units` has multiple entries with `dimension_names`, `generate-report` switches to evaluation report mode:

- Per-dimension findings with dimension-prefixed IDs (CON-xx, VER-xx, etc.)
- Cross-dimensional patterns section
- Core finding that explains the most findings across dimensions
- Recommendations organised by dimension

This is the same report structure currently produced by prism-evaluate's `consolidate-report` — absorbed into prism's generate-report as a conditional mode triggered by the presence of `dimension_names` in the analysis units.

## 5. What Happens to `prism-evaluate`

### Before

```
prism-evaluate:
  scope-definition      → Elicitation + dimension derivation
  dimension-planning    → Map dimensions to prism config
  execute-analysis      → Trigger prism N times
  consolidate-report    → Cross-dimensional synthesis
  deliver-results       → Present results
```

### After

Every responsibility is absorbed:

| prism-evaluate component | Absorbed into |
|---|---|
| scope-definition (elicitation) | prism's `select-mode` elicitation step |
| dimension-planning (dimension derivation) | prism's `plan-analysis:parse-prompt` + `map-dimensions` |
| dimension-planning (target survey) | prism's `plan-analysis:parse-prompt` target read |
| dimension-planning (dimension-to-lens mapping) | prism's `plan-analysis:map-dimensions` |
| execute-analysis (multi-run loop) | prism's `execute-plan` unit dispatch loop |
| consolidate-report (cross-dimensional synthesis) | prism's `generate-report` evaluation mode |
| deliver-results | prism's `deliver-result` (already exists) |

**prism-evaluate can be deprecated.** Its functionality is fully available via a structured `analysis_prompt` passed to prism. The caller that previously used prism-evaluate now calls prism directly with a multi-dimensional prompt.

### Migration path for existing prism-evaluate callers

```
# Before (prism-evaluate)
workflow: prism-evaluate
variables:
  evaluation_description: "Evaluate for consistency and feasibility"
  target_path: /path/to/proposal.md
  output_path: /output/

# After (prism with structured prompt)
workflow: prism
variables:
  analysis_prompt: "Evaluate for consistency and feasibility"
  target: /path/to/proposal.md
  output_path: /output/
```

## 6. Implementation Phases

### Phase 1: Elicitation + structured prompt
- Add `analysis_prompt` and `prompt_was_elicited` variables to workflow.toon
- Add `elicit-goals` step to select-mode with `analysis_prompt notExists` gate
- Add `confirm-prompt` checkpoint (gated on `prompt_was_elicited`)
- Save generated prompts to `{planning_folder}/analysis-prompt.md`
- No downstream changes yet — elicitation produces a prompt, existing plan-analysis consumes it as the analytical goal

### Phase 2: parse-prompt + map-dimensions in plan-analysis
- Add `parse-prompt` and `map-dimensions` protocol steps to plan-analysis
- Extract multi-dimensional analysis plans from structured prompts
- Gate on `analysis_prompt` existence — fall through to existing logic when absent
- Single-dimension prompts produce one analysis_unit (backward-compatible)

### Phase 3: execute-plan activity + per-unit dispatch
- Add `execute-plan` activity between `select-mode` and mode-specific activities
- Move the unit iteration loop from `structural-pass` to `execute-plan`
- `structural-pass` and other mode activities become single-unit executors
- Multi-dimensional prompts produce multiple analysis_units with different modes

### Phase 4: Generate-report evaluation mode + prism-evaluate deprecation
- Extend `generate-report` to detect multi-dimensional units and produce evaluation-format reports
- Deprecate `prism-evaluate` — mark as superseded by prism with structured prompt
- Migrate any existing prism-evaluate callers

### Phase 5: Remove caller-side configuration variables
- Remove `pipeline_mode`, `selected_lenses` from the caller-facing contract
- Keep them as internal state variables set by plan-analysis
- Breaking change — requires all callers to migrate to `analysis_prompt`

---

## 7. The Invariant

**prism owns all analytical decisions.** The structured prompt is a description of what to understand, expressed in the caller's domain language. It never references prism internals (modes, lenses, resources, indices). When prism gains new modes, all callers benefit automatically because the mode-selection logic is inside prism, not distributed across callers.
