# Evaluation Report — Schema Effectiveness, Technique Markdown Structure & Progressive-Disclosure Economics

Date: 2026-07-03. Scope: read-only review per `.engineering/artifacts/templates/schema-technique-disclosure-review-prompt.md`. Supporting evidence in this folder: [field-ledger.md](field-ledger.md) (1a), [corpus-consumption-audit.md](corpus-consumption-audit.md) (1b), [authoring-friction-guards.md](authoring-friction-guards.md) (1c), [technique-structural-census.md](technique-structural-census.md) (2a), [probe-transcripts.md](probe-transcripts.md) (2b), [contract-seam-check.md](contract-seam-check.md) (2c), [payload-measurements.md](payload-measurements.md) (3a–3c).

**Method:** field-by-field trace of all six schemas into `src/` consumers; consumption counts across all 14 workflows (104 activities, 673 steps); guard-script and remediation-history reconstruction; structural census of 206 technique files; six fresh-agent comprehension probes on real composed payloads; instrumented live walks of `work-package` (full, 252 tool calls) and `ponytail` through the real server with exact per-response byte accounting.

---

## Executive summary

1. **The schemas describe far more machine than exists.** The server is a state ledger + payload composer: it enforces checkpoints, session identity, and load-time structure, and applies exactly one declarative effect (`setVariable`). Everything else — every condition, gate, loop bound, action verb, trigger, variable type/default — is agent-interpreted or entirely inert. The schema's biggest interpretive hazard is that inert fields *look* executable.
2. **Technique markdown works.** Six of six probes reconstructed protocols faithfully across every shape; the de-facto template is near-universal (95/179 exact). The failures are at the *seams*, not in the structure: inherited required-but-unused inputs, unstated input provenance, dead declared outputs, unspecified output mechanics.
3. **Progressive disclosure is directionally right but pays two hidden taxes** that dwarf the per-call envelope it was tuned around: a ~15.5–17.8k-char inherited bundle re-delivered byte-identically in **every** `get_activity` (20 % of a full work-package walk; 89 % of every ponytail `get_activity`), and byte-identical technique refetches (39 % of fetch calls, 49 % of technique-delivery volume). Both are fixable without touching disclosure granularity.

## 2. Ledger verdicts

Full inventories: [field-ledger.md](field-ledger.md), [corpus-consumption-audit.md](corpus-consumption-audit.md), [technique-structural-census.md](technique-structural-census.md). Verdicts: **keep / clarify** (rename, re-document, or enforce) / **merge** / **retire-candidate** (flag for a future major version; no breaking change now).

**Workflow schema:** keep `id/version/title/description/tags/rules/techniques/variables(name,description)/initialActivity/activities`. Clarify `initialActivity` (warn-only reality). Retire-candidates: `author` (inert, unread), `variables[].type/defaultValue/required` **unless** backlog B7 (seed defaults engine-side) is taken — then `type`/`defaultValue` become ENFORCED and stay.
**Activity schema:** keep `id/version/name/description/steps/transitions/outcome/required/techniques`. Clarify `outcome` (server never reconciles it against manifests — say so) and activity-file `rules` vs workflow `rules.activity` (same name, different delivery — merge into one surface). Merge `when`/`condition` (pick the structured form; give it the dismissal semantics; 17 vs 233 uses). Retire-candidates: `triggers[]`+`passContext` (inert AND 3 corpus uses), `decisions[]` (7 uses; transitions won), authored `artifacts[]` (0 uses, guard-forbidden — delete from schema now, B4), `step.required`, `step.actions[]` — either retire or implement `set` (see B7); `emit`/`message` verbs, per-action `condition`.
**Checkpoint:** keep `message/options/effects/defaultOption/autoAdvanceMs`. Clarify `blocking` (inert; either enforce or drop) and document that `transitionTo`/`skipActivities` are **not** engine-applied. Retire-candidate: `skipActivities` (0 uses, write-only server-side).
**Loop:** keep `loopType/variable/over/steps/maxIterations`(as documented-advisory). Retire-candidates: `breakCondition` (0 uses, inert), `doWhile` (2 wfs — watch).
**Condition:** keep `simple/and/or/not`, `==/!=`, `exists/notExists`. Retire-candidates: `>=`/`<=` (0 uses); document that **all** evaluation is agent-side. Fix generated-schema recursion (B5).
**Technique markdown:** keep the modal template (Capability / Inputs / Outputs / Protocol / Rules, `###` ids, `####` components) — it demonstrably transmits. Clarify: inherited-input presentation (B2), `{$var}` sigil (document in TECHNIQUE authoring guide; currently work-package folklore), Outputs-before-Protocol ordering (lint, B9). Retire-candidates: `Initial`/`Final` wrapping (0 uses — keep code dormant or delete at next major), `outputs[].artifact.action` (unreachable from authored content).

## 3. Disclosure-policy verdict

**Keep per-step `get_technique` disclosure as the default. The granularity is not the problem — the repetition is.** Measured on the full work-package walk (287k tokens total):

- The true per-call envelope is ~570 chars (12 % of a mean fetch) — cheap. Lazy fetching breaks even against per-activity bundling when ≥14 % of bundled techniques would go unused; corpus gating (~18–22 % of technique steps) sits right at that line, and large fail-path techniques (`analyze-failure`, 4.6k, fetched only on failure) are exactly what bundling would waste.
- Per-activity bundling would cut technique-delivery volume 39 % and tool calls 64 % — but **almost all of that saving is deduplication of repeats, not envelope removal**, and it forfeits the per-step attention anchor plus read-ahead protection that the design record names as the reason step techniques are deliberately not bundled into `get_activity`. It also delivers gated-step content that never executes.
- **Delta mode gets the same token saving without the fidelity trade:** full content on first fetch per session context, ~200-char reference on identical refetch → −47 % technique delivery (391k vs 743k chars), granularity unchanged.
- The larger prize is F1: the inherited techniques+rules bundle repeated in every `get_activity`. Reference-after-first-delivery converts the walk to ≈145k tokens (**−50 % overall**) with zero change to what an agent may read.
- **Context-model caveat (decisive for defaults):** repetition is only waste when the *same context* receives it twice. In disposable-worker sessions (prism-style, one worker per activity) the re-delivered bundle is load-bearing — each fresh worker needs it. Therefore both fixes must be **opt-in per session or per call** (declared context mode / `bundle: reference` parameter), not a changed default.
- Where per-step disclosure genuinely loses is **turn overhead**: 161 fetch round-trips per walk (~3–8 min pure protocol latency). The measured hybrid — bundle techniques ≤4k unique chars into `get_activity`, lazy-fetch the rest — removes 75 % of fetch calls for +19 % technique-delivery chars vs delta-only, and keeps large/gated techniques lazy. Worth offering as an **activity-declared opt-in**, not a default.
- Fidelity ledger (3c): the 39 % refetch rate partly *is* fidelity (fresh attention per binding site); delta mode therefore needs an explicit full-content escape (context summarized/stale). Bundling recommendations above stay opt-in for the same reason. Today no fidelity failure is even observable server-side (F12); B8 adds the missing signal before any bolder bundling default is considered.

## 5. Non-goals honored

No recommendation above replaces the MCP server, session model, YAML+JSON-Schema definitions, markdown techniques, or agent-side interpretation; every item is a field addition/removal flag, payload-shaping option, renderer/validation improvement, lint, or documentation fix, and every default-behavior change is opt-in. Two findings that *could* motivate architectural discussion are explicitly deferred: (a) the server's wholesale reliance on agent-side condition/loop evaluation (the engine's `evaluateCondition` is runtime-dead) — moving evaluation server-side would be a different execution model, out of scope here and adjacent to the framework-vs-skills comparison (see `.engineering/artifacts/templates/framework-vs-skills-review-prompt.md`); and (b) the observation that guards exist chiefly because contracts span YAML and markdown — unifying the two formats would be a re-architecture, not an increment, and is likewise deferred to that review.
