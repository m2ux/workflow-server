## Decision

The documentation system is delivered as a **hand-authored static HTML site** (semantic HTML + one shared stylesheet + inline SVG diagrams) instead of an MkDocs Material frame. Direction set by the maintainer on 2026-07-07, guided by [Using Claude Code: the unreasonable effectiveness of HTML](https://claude.com/blog/using-claude-code-the-unreasonable-effectiveness-of-html).

## Rationale

| Driver | MkDocs Material (planned) | Authored HTML (adopted) |
|--------|---------------------------|-------------------------|
| Diagrams | Mermaid/plugin-bound, generic styling | Bespoke inline SVG per concept (model hierarchy, state machine, checkpoint swimlanes, dispatch flow, binding data-flow) |
| Information design | One theme for all pages | Per-page layout tuned to the content (tabs, cards, annotated transcripts) |
| Toolchain | Python + `mkdocs-material` — the repo's first Python dependency | None beyond Node already present; site data generated by a `tsx` script like the existing `generate-schemas.ts` |
| API freshness | Hand-maintained markdown | `api/` pages regenerated from `src/` tool registrations and `schemas/*.json` each build, with a staleness guard |
| Requirements | — | Maintainer requirements: diagram-rich, accessible language, user guide / technical specs / API usage sections, continuous deployment |

## What carries over from the original plan (still valid)

- **Zero link-rot constraint (R-4/IA-2):** all existing markdown files stay in place, unrenamed; the HTML site is a rendered layer over them, so the ~50 inbound `.md` links are untouched.
- **Markdown stays canonical.** HTML pages link back to their canonical `.md` source. The highest-drift-risk content (API/tool/schema reference) is machine-generated; narrative pages are regenerated on demand and reviewed like any PR.
- **Section grouping (nav mapping):** Getting Started + Agent Guidance → `site/guide/`, Architecture (with `docs/architecture.md` as hub, IA-4) → `site/specs/`, API Reference → `site/api/`.
- **Drift-guard philosophy (G8):** `mkdocs build --strict` is replaced by repo-idiomatic checks — a site link/anchor checker and a generated-fragment staleness check (regenerate + diff, matching the existing `check:*` script pattern).
- **Out of scope, unchanged:** ADRs (remain solely under `.engineering/`, DP-6/Q-IMPL-1), FAQ/troubleshooting (Q-IMPL-5), `src/`/`schemas/` behavior changes, file renames.

## Layer C checkpoint — resolved

The original plan deferred the deploy automation pending explicit approval (*Decision Checkpoint: Layer C*, `file-sensitivity-cicd-approval`). The maintainer's 2026-07-07 direction explicitly requests continuous deployment, resolving the checkpoint **approved**, with the deployment target **GitHub Pages via Actions artifact deployment** (`actions/deploy-pages`). The workflow is Node-only — the Python-in-CI concern from the original checkpoint does not arise. One-time manual step remains: enable Pages (source: GitHub Actions) in repository settings. R-3 stands: no multi-version docs tooling.

## Accessibility commitments (applies to every page)

Semantic landmarks and heading hierarchy; every SVG carries `role="img"` with `<title>`/`<desc>` and its information restated in nearby prose; color never the sole differentiator; WCAG AA contrast; plain-language register in the guide section written for a reader new to MCP.
